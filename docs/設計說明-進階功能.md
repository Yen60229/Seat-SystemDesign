# 進階功能設計：認證、公告、UI 優化

分支：`feature/auth-ui`（`main` 保留為原始可交版本，不受影響）

本文件接續 [設計說明.md](設計說明.md)，在原員工座位系統上新增登入認證、忘記密碼、
公告欄與面向 35–55 歲使用者的 UI 優化。實作以**嚴謹安全規格**與 **TDD（測試先行，
包含邊界情境）** 為原則。

---

## 1. 範圍

| 功能 | 說明 |
|---|---|
| JWT 登入 | 員編 + 密碼登入，發放 JWT（httpOnly cookie） |
| 角色權限 | ADMIN（人資）可發公告、排座位；EMPLOYEE 唯讀 |
| 忘記密碼 | 索取 6 碼驗證碼（模擬寄送）→ 驗證 → 重設密碼 |
| 公告欄 | 登入後首頁顯示；ADMIN 可新增/編輯/刪除 |
| 座位擬真 | 各樓層不同格局（座標化），含走道、主管角落位 |
| UI 優化 | 大字體、高對比、明確流程，適合中高齡使用者 |

預設帳號 = 員編；預設密碼 = 出生年月日 + 身分證後 4 碼，
**僅以 bcrypt 雜湊存入 DB，原始個資不入庫**。開發驗收用的種子帳密集中記錄於
[seed-passwords.md](seed-passwords.md)。

## 2. 安全規格（嚴謹版）

| 面向 | 做法 |
|---|---|
| 密碼儲存 | BCrypt（cost 12），不可逆雜湊 |
| Token 儲存 | JWT 放 **httpOnly + Secure + SameSite=Strict** cookie，JS 讀不到，防止 XSS 竊取 Token |
| Token 內容 | HS256 簽章，內含 `sub`(員編)、`role`、`exp`(2 小時)；驗章 + 驗到期 |
| CSRF | SameSite=Strict 防止跨站請求自動帶入 Cookie；異動 API 僅同源可用 |
| 帳號列舉防護 | 登入錯誤一律回「帳號或密碼錯誤」；忘記密碼無論帳號是否存在都回相同訊息 |
| 驗證碼 | 6 碼數字，**雜湊後儲存**、10 分鐘到期、**單次使用**、最多嘗試 5 次 |
| 速率限制 | 登入 / 索取驗證碼 以員編為單位限流（記憶體計數，正式環境改用 Redis） |
| 授權 | ADMIN 才能異動公告 / 座位；未登入回 401、權限不足回 403 |
| 密碼強度 | 重設新密碼至少 8 碼且含英數 |
| SQL Injection | 沿用原有規範：全程 Stored Procedure + 參數綁定 |
| XSS | Vue 預設轉義、不使用 `v-html`；公告內容以純文字呈現 |

`SP_GET_ANNOUNCEMENTS` / `SP_CREATE_ANNOUNCEMENT` / `SP_UPDATE_ANNOUNCEMENT` / `SP_DELETE_ANNOUNCEMENT`、

- **12F 主管樓層**：座位較少 + 角落 `MANAGER` 位

## 4. REST API

| Method | Path | 權限 | 說明 |
|---|---|---|---|
| POST | `/api/auth/login` | 公開 | 登入，成功種 httpOnly cookie |
| POST | `/api/auth/logout` | 登入 | 清除 cookie |
| GET | `/api/auth/me` | 登入 | 取得目前使用者（員編、姓名、角色） |
| POST | `/api/auth/forgot-password` | 公開 | 索取驗證碼（模擬寄送） |
| POST | `/api/auth/reset-password` | 公開 | 驗證碼 + 新密碼 → 重設 |
| GET | `/api/announcements` | 登入 | 公告清單 |
| POST/PUT/DELETE | `/api/announcements/**` | ADMIN | 公告 CRUD |
| GET | `/api/floors`、`/api/floors/{n}/seats`、`/api/employees` | 登入 | （沿用） |
| POST | `/api/seats/batch` | ADMIN | 座位異動（沿用） |

模擬寄送：`forgot-password` 在開發模式於回應與後端 log 附帶驗證碼，前端以明顯測試提示框呈現。

## 6. 前端（UI 優化 + 擬真座位）

- 路由：`/login`、`/forgot-password`、`/`（首頁：公告 + 座位）
- 守衛：未登入導向 `/login`；角色決定可否看到管理操作
- 擬真座位：依 `POS_ROW/POS_COL` 在平面圖定位，空位處留白成走道；`MANAGER` 位視覺加強
- UI 原則：基準字級 ≥16px、主按鈕大且高對比、狀態色彩明確（綠/紅/黃 + 文字標籤輔助，不單靠顏色）、流程一頁一焦點

## 7. 測試策略（TDD，包含邊界情境）

每階段先寫測試再寫實作。各模組 edge case 清單：

**密碼雜湊 / AuthService**
- 正確密碼通過、錯誤密碼拒絕
- 帳號不存在 → 與密碼錯誤回相同錯誤（防列舉）
- 員編大小寫處理一致
- 連續失敗達上限 → 限流錯誤

**JwtService**
- 簽發後可驗回相同 claims
- 過期 token 被拒
- 簽章被竄改被拒
- 格式錯誤 / 空 token 被拒
- 角色 claim 正確帶入

**PasswordResetService**
- 索取驗證碼：帳號不存在不洩漏（回應一致）、重複索取使舊碼失效
- 驗證：碼正確且未過期未使用 → 通過
- 過期碼被拒、已用碼被拒、錯誤碼被拒
- 嘗試次數超過上限 → 鎖定
- 重設成功後該碼標記已用、舊碼無法再用
- 新密碼太弱被拒

**授權（Controller/Security）**
- 未登入呼叫受保護 API → 401
- 一般員工呼叫 ADMIN API → 403
- ADMIN 呼叫 → 通過

**AnnouncementService**
- 標題空白 / 超長 → 驗證錯誤
- 內容含 HTML/script 字串 → 安全儲存與呈現（不執行）
- 編輯 / 刪除不存在的公告 → 明確錯誤

**座位（沿用 + 擬真）**
- 既有交易還原、並發撞鍵測試沿用
- 座標渲染：同樓層座位座標不重疊
